# Wireframe: Getting Started (proposed)

Low-fi structural wireframe for the new canonical Getting Started page. This replaces the current `getting-started.md` AND `quick-start.md` (which today live as two competing pages).

Annotations in [brackets] are wireframe notes, not page copy.

## Page-level design principles

Five rules this wireframe enforces:

1. **One linear path through.** No choose-your-own-adventure at step zero.
2. **5 minutes to first success.** Time anchor in the hero.
3. **Surface the verification gate.** The single biggest activation killer becomes explicit.
4. **Define what success looks like.** Every step ends with "you should see X."
5. **Branching at the end, not the start.** Three next-step cards offered only after first success.

## Annotated decision table

| Decision | Replaces today | Why |
|---|---|---|
| One linear path | Quickstart options A-E | Choose-your-own-adventure at step zero forces decisions users can't yet make. Stripe and Vercel both moved away from this pattern years ago. |
| Verification gate is explicit | Hidden in FAQ today | The #1 activation killer in the current funnel. Marketing says "free GPU," UI prompts for credit card, the unlock path is undocumented in Getting Started. |
| sentence-transformers embedding demo as first example | "Start building" link table with 12 destinations | Runs entirely on free CPU. No API keys (zero auth friction). Demonstrates real AI work, not just an API forward. Embeddings are the primitive behind RAG and search, so it lands as immediately relevant. |
| Time anchors per section | Nothing today | "~1 min / ~1 min / ~3 min" sets commitment expectation. Drives completion rate. |
| Success state defined per step | Not defined today | Users don't know if they succeeded. Today's docs end mid-stream with no "you should see X." |
| 3 next-step cards | Two ~10-cell link tables | Decision fatigue. Three is the upper bound on "I can pick one." |
| Why Lightning demoted to footer | Equal-weight card on current Overview | Marketing-style positioning belongs in Concepts, not first-touch. |
| New section: Concepts | Mixed into Overview today | "Overview" should be navigational, not editorial. Concepts is where the "what is this" content lives. |

## Open questions

1. **What % of new accounts use a work/.edu email vs phone vs personal?** This determines whether the GPU verification table should default-show the email path or the phone path.
2. **What's the current funnel from `/docs/platform/getting-started` to a user running their first job?** Without this we're optimizing blind.
3. **Is sentence-transformers the right first example, or should it be PyTorch Lightning MNIST (the framework Lightning is famous for), or a LitAI hosted-model call (uses Lightning's own product)?** Current pick: sentence-transformers. Reasoning: zero auth friction, runs on free CPU, demonstrates real AI work. Counter-argument: doesn't use any Lightning-branded product, so the example doesn't sell the platform's stack. Worth user-testing all three.
4. **Do we want an "I just want the API ref" jump-link at the top?** Stripe does this. Worth A/B testing.
